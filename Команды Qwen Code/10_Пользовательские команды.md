# 10_Пользовательские команды

> Создание собственных команд для автоматизации часто используемых действий

---

## Что такое пользовательские команды?

Пользовательские команды позволяют сохранять часто используемые промпты и инструкции как именованные команды, которые можно быстро вызывать через `/`.

**Пример:**
```
# Вместо ввода длинного текста:
"Проанализируй код на наличие багов, проверь стиль кода, добавь комментарии..."

# Создайте команду:
/review

# И используйте:
/review @src/main.py
```

---

## Расположение файлов команд

### Глобальные команды

**Путь:** `~/.qwen/commands/`

**Доступны:** Во всех проектах и сессиях.

**Примеры путей:**
- **Windows:** `C:\Users\<Имя>\.qwen\commands\`
- **Linux/Mac:** `~/.qwen/commands/`
- **Mac (полный):** `/Users/<Имя>/.qwen/commands/`

---

### Проектные команды

**Путь:** `<корень проекта>/.qwen/commands/`

**Доступны:** Только в текущем проекте.

**Пример:**
```
D:\Projects python\Help\.qwen\commands\
```

---

## Создание команд

### Базовая структура

**Файл команды:** `<имя>.md`

**Содержимое:**
```markdown
<Ваш промпт или инструкция>
```

**Пример:**
Файл `review.md`:
```markdown
Проанализируй этот код:
1. Найди потенциальные баги
2. Проверь соответствие стилю кода
3. Предложи улучшения
4. Добавь комментарии к сложным местам
```

---

### Именование файлов

**Правило:** Символ `/` в пути заменяется на `:` в имени команды.

**Примеры:**

| Путь файла | Имя команды |
|-----------|-------------|
| `review.md` | `/review` |
| `tests/unit.md` | `/tests:unit` |
| `code/refactor.md` | `/code:refactor` |
| `git/commit.md` | `/git:commit` |
| `deep/nested/path.md` | `/deep:nested:path` |

## Примеры пользовательских команд

### 1. Команда для код-ревью

**Файл:** `~/.qwen/commands/review.md`

```markdown
Проведи код-ревью предоставленного кода:

🔍 **Анализ:**
1. Найди потенциальные баги и уязвимости
2. Проверь соответствие best practices
3. Оцени читаемость и поддерживаемость

📝 **Рекомендации:**
1. Предложи конкретные улучшения
2. Покажи примеры исправленного кода
3. Укажи на проблемные места

⭐ **Оценка:**
- Общее качество: 1-10
- Критичность проблем: низкая/средняя/высокая
```

**Использование:**
```
/review @src/auth.py
```

---

### 2. Команда для написания тестов

**Файл:** `~/.qwen/commands/test.md`

```markdown
Напиши комплексные тесты для этого кода:

✅ **Требования:**
1. Покрой все публичные функции
2. Включи тесты на граничные случаи
3. Добавь тесты на ошибки
4. Используй mock для внешних зависимостей

📋 **Формат:**
- Названия тестов должны описывать поведение
- Каждый тест должен быть независимым
- Добавь комментарии к сложным тестам

🎯 **Покрытие:**
- Стремись к 80%+ покрытию
- Приоритет: важная бизнес-логика
```

**Использование:**
```
/test @src/calculator.py
```

---

### 3. Команда для документирования

**Файл:** `~/.qwen/commands/document.md`

```markdown
Создай документацию для этого кода:

📖 **Docstrings:**
1. Добавь docstring к каждой функции
2. Опиши параметры и возвращаемые значения
3. Приведи примеры использования
4. Укажи возможные исключения

📄 **README:**
1. Краткое описание модуля
2. Основные функции
3. Примеры кода
4. Зависимости

🔧 **Формат:**
- Google style или NumPy style
- На русском языке
- С примерами
```

**Использование:**
```
/document @src/api/
```

---

### 4. Команда для рефакторинга

**Файл:** `~/.qwen/commands/refactor.md`

```markdown
Выполни рефакторинг этого кода:

🎯 **Цели:**
1. Улучши читаемость
2. Устрани дублирование (DRY)
3. Разбей большие функции
4. Упрости сложную логику

📐 **Принципы:**
- SOLID
- KISS (Keep It Simple, Stupid)
- YAGNI (You Ain't Gonna Need It)

⚡ **Важно:**
- Сохрани функциональность
- Не сломай существующие тесты
- Добавь комментарии к изменениям
```

**Использование:**
```
/refactor @src/legacy/old_module.py
```

---

### 5. Команда для Git-коммитов

**Файл:** `~/.qwen/commands/git/commit-message.md`

**Команда:** `/git:commit-message`

```markdown
Проанализируй изменения и создай сообщение для коммита:

📋 **Формат:**
```
<тип>(<область>): <краткое описание>

<развёрнутое описание>

<ссылки на issues>
```

🏷️ **Типы:**
- feat: новая функция
- fix: исправление бага
- docs: документация
- style: форматирование
- refactor: рефакторинг
- test: тесты
- chore: прочее

📝 **Требования:**
- Заголовок до 50 символов
- Тело до 72 символов в строке
- Используй повелительное наклонение
```

**Использование:**
```
!git diff --staged
/git:commit-message
```

---

### 6. Команда для объяснения кода

**Файл:** `~/.qwen/commands/explain.md`

```markdown
Объясни этот код простым языком:

🎓 **Для кого:**
- Начинающий разработчик
- Не знаком с этим языком/фреймворком

📖 **Структура объяснения:**
1. Общая цель кода (1-2 предложения)
2. Основные компоненты и их функции
3. Поток данных (как работает)
4. Ключевые концепции

💡 **Дополнительно:**
- Аналогии из реальной жизни
- Визуальные описания
- Примеры входных/выходных данных
```

**Использование:**
```
/explain @src/complex_algorithm.py
```

## Продвинутое использование

### Переменные в командах

Некоторые версии поддерживают переменные:

**Синтаксис:**
```markdown
Проанализируй код с фокусом на: {{focus}}

Файлы: {{files}}
```

**Использование:**
```
/review --focus=безопасность @src/auth.py
```

---

### Шаблоны с условиями

**Пример:**
```markdown
{% if language == "python" %}
Проверь соответствие PEP 8
{% elsif language == "javascript" %}
Проверь соответствие ESLint
{% endif %}
```

---

### Команды с несколькими файлами

**Файл:** `~/.qwen/commands/compare.md`

```markdown
Сравни предоставленные файлы:

📊 **Анализ:**
1. Найди различия в реализации
2. Оцени преимущества каждого подхода
3. Выяви общие паттерны
4. Предложи лучший вариант

📋 **Формат:**
- Таблица сравнения
- Плюсы и минусы
- Рекомендации
```

**Использование:**
```
/compare @v1/api.py @v2/api.py
```

## Советы по созданию команд

### ✅ Лучшие практики

1. **Одно назначение — одна команда**
   ```
   /review      ✅ (только ревью)
   /everything  ❌ (всё подряд)
   ```

2. **Используйте понятные имена**
   ```
   /review      ✅
   /r           ❌ (неясно)
   /code-review ✅ (тоже хорошо)
   ```

3. **Структурируйте промпты**
   ```markdown
   # Цель
   # Требования
   # Формат вывода
   # Примеры
   ```

4. **Добавляйте примеры использования**
   ```markdown
   # Пример:
   # /review @src/main.py
   ```

5. **Версионируйте команды**
   ```
   review-v1.md
   review-v2.md
   ```

---

### ⚡ Продуктивность

**Создавайте команды для:**
- Повторяющихся задач
- Сложных инструкций
- Стандартных рабочих процессов
- Командной работы

**Пример экономии времени:**
```
# Без команды (5 минут ввода):
"Проанализируй код, найди баги, проверь стиль, добавь комментарии, 
предложи улучшения, оцени качество..."

# С командой (1 секунда):
/review
```

## Решение проблем

### Команда не работает

**Проблема:** `/mycommand` не распознаётся.

**Проверка:**
1. Файл существует в правильной папке?
2. Имя файла соответствует имени команды?
3. Файл имеет расширение `.md`?
4. Перезапустили Qwen Code?

**Решение:**
```bash
# Проверка (Linux/Mac)
ls ~/.qwen/commands/

# Проверка (Windows)
dir %USERPROFILE%\.qwen\commands\
```

---

### Команда выполняется неправильно

**Проблема:** Qwen не следует инструкции.

**Решение:**
1. Уточните формулировку в файле команды
2. Добавьте больше деталей
3. Приведите примеры ожидаемого вывода
4. Проверьте синтаксис Markdown

---

### Конфликт имён

**Проблема:** Две команды с одинаковым именем.

**Решение:**
1. Проектные команды имеют приоритет над глобальными
2. Переименуйте одну из команд
3. Используйте подпапки для пространства имён
   ```
   git/commit.md  → /git:commit
   docs/commit.md → /docs:commit
   ```
